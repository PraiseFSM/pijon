## A. Duplicated logic → shared helpers (8.D2)

### A1. ✅ `kindFillColor` / `kindStrokeColor` in FurnitureEditor duplicate `fillForFurniture` / `strokeForFurniture` in render.ts — **DONE (8.D2)**

- **Files:**
  - `src/ui/editors/FurnitureEditor.tsx:1164–1178` — `kindFillColor(kind)` and `kindStrokeColor(kind)`
  - `src/ui/canvas/render.ts:210–229` — `fillForFurniture(f)` and `strokeForFurniture(f)`
- **Situation:** Both switch on `FurnitureKind` and return the same color constants from `colors.ts`. The FurnitureEditor copies were introduced to make the live-drag preview look identical to the base pass. They are not exported.
- **Resolution:** Renamed `fillForFurniture`/`strokeForFurniture` in `render.ts` to `fillForFurnitureKind`/`strokeForFurnitureKind` (kind-based, exported). `drawSingleFurniture` now calls them with `f.kind`. FurnitureEditor imports and uses the shared helpers; local `kindFillColor`/`kindStrokeColor` removed. Unused color imports cleaned up. New test file: `src/test/iteration8_clusterD_furnitureColors.test.ts` (9 tests).

### A2. SKIPPED — `paintFurnitureRect` in FurnitureEditor partially duplicates `drawSingleFurniture` in render.ts

- **Files:**
  - `src/ui/editors/FurnitureEditor.tsx:1188–1225` — `paintFurnitureRect`
  - `src/ui/canvas/render.ts:238–298` — `drawSingleFurniture`
- **Situation:** Both look up the image URL, call `getImage`, fall back to fill-color, and call `ctx2d.drawImage / fillRect / strokeRect`. Differences: `paintFurnitureRect` accepts alpha + arbitrary rect; `drawSingleFurniture` computes the rect from the Furniture object and CanvasView.
- **Skip reason:** Legitimate behavioral differences — `paintFurnitureRect` uses `globalAlpha` for ghost transparency and takes a pre-computed rect (drag position); `drawSingleFurniture` takes a `Furniture` object and applies a lock tint. After A1, the color logic is shared. Merging the structural differences would add option-params complexity that exceeds the benefit. The color duplication (the real risk) is resolved by A1.

### A3. SKIPPED — Context-menu open/close logic duplicated across StudentEditor

- **Files:** `src/ui/editors/StudentEditor.tsx` — context menu show/hide bridge
  (`showContextMenuCallback`, `registerContextMenuCallback`, `setShowContextMenu`) are module-level state + a React-effect bridge that couples the canvas event handler to the React component. The same pattern is used in two different places in the file.
- **Skip reason:** This is a structural refactor (lifting state to Zustand or a React context), not a simple deduplication. Changing the singleton pattern risks test fragility and behavior differences. Deferred to a dedicated refactor pass (see C3).

## C. Undertested / hard-to-maintain areas (8.D4 / 8.D5)

### C1. FurnitureEditor.tsx (2014 lines) is very large

- **File:** `src/ui/editors/FurnitureEditor.tsx`
- **Observation:** The file contains the editor logic, the ghost-ring resize UI, the custom furniture import flow, the live-drag preview renderer, the granularity ghost-fix overlay, and the full palette SidePanel. These are distinct responsibilities.
- **Proposed split (if clearer):**
  - `FurnitureEditor.tsx` — EditorMode glue, activate/deactivate, event handlers, overlay orchestration
  - `FurnitureEditorOverlay.ts` — `paintFurnitureRect`, `kindFillColor`, `kindStrokeColor`, ghost-fix paint helpers
  - `FurnitureSidePanel.tsx` — palette items, custom import form (already a sub-component inside the file)
  - `FurnitureToolbar.tsx` — New / Clear / Granularity / gridColor / BG controls (already a sub-component)

### C2. StudentEditor.tsx (2471 lines) is very large

- **File:** `src/ui/editors/StudentEditor.tsx`
- **Observation:** Contains roster, seating overlay, assigner mode, preference links, context menu, drag-to-seat, and roster-drag-to-seat. All share module-level state.
- **Proposed split:**
  - `StudentEditor.tsx` — EditorMode glue + module-level state
  - `StudentEditorOverlay.ts` — `paintOverlay` and all overlay helpers
  - `StudentRosterPanel.tsx` — roster list, CSV import, add-student form (already a sub-component)
  - `StudentPrefDetail.tsx` — preference detail panel (already a sub-component)

### C3. Module-level singleton state in both editors

- **Files:** `StudentEditor.tsx`, `FurnitureEditor.tsx` — both use module-level `let` variables for transient state (selectedId, dragState, neighborPreviewFid, etc.)
- **Observation:** Module-level state is cleared on `activate()` / `deactivate()`, but leaks between test runs unless each test explicitly activates/deactivates the editor. This is fragile.
- **Proposed fix:** Wrap transient state in a class or a factory function so each "instance" has its own state. The EditorMode interface would need to return an instance from a factory rather than being a singleton object.

### C4. `src/state/hooks.ts` has only one export

- **File:** `src/state/hooks.ts`
- **Observation:** Only `useSeatingIssues` lives here. As more safe hooks are added they should go here too. Low maintenance risk today, but note it as the canonical home for Zustand safe-hook wrappers.

### What is intentionally deferred and why

**C1 / C2 — Splitting the large editor files:** Both editors carry extensive
module-level singleton state (`selectedId`, `dragState`, `markerFirstFid`,
`cachedGraph`, `granFixPendingG`, etc.) that is read and written across event
handlers, React effect callbacks, and `paintOverlay`. These variables must be
co-located with the code that writes them to avoid cross-file circular state.
Splitting would require either:
  - Moving the state to Zustand (a product change — affects undo/redo, test isolation),
  - Or creating a shared mutable state object (swaps the problem for a different one).

Recommended future structure when ready to invest in the refactor:

```
src/ui/editors/furniture/
  FurnitureEditorState.ts       # module-level mutable singleton (granFix, dragState, selectedId, ...)
  FurnitureEditorOverlay.ts     # paintFurnitureRect, paintOverlay (imports State)
  FurnitureSidePanel.tsx        # palette items, custom import form
  FurnitureToolbar.tsx          # New / Clear / Granularity / gridColor / BG controls
  index.ts                      # re-exports FurnitureEditor EditorMode instance

src/ui/editors/student/
  StudentEditorState.ts         # module-level mutable singleton (dragState, markerFirstFid, cachedGraph, ...)
  StudentEditorOverlay.ts       # paintStudentOverlay, paintPreferenceLinks, hasViolation
  StudentRosterPanel.tsx        # roster list, CSV import, add-student form
  StudentPrefDetail.tsx         # preference detail panel
  index.ts                      # re-exports StudentEditor EditorMode instance
```

Migration steps:
1. Create the `state` module with all module-level `let` variables. Export them
   via getter/setter functions so TypeScript tracks mutability.
2. Move paint functions to overlay modules (they import from state, not vice versa).
3. Split panel components last (they only read store state, not module-level state).
4. Run tests after each step — the existing test suite (50 files, 1719 tests) gives
   high confidence if all pass after each extraction.

**C3 — Module-level singleton state is fragile across tests:** See above. The underlying
issue is that `activate()` / `deactivate()` are the only reset points. Future work:
factory-function pattern so each editor "instance" has its own state object.

**E1 — src/test/ directory is flat:** The flat naming is iteration-oriented rather than
feature-oriented. Proposed grouping in the original E1 observation above is still valid.
Reorganizing requires updating Vitest config (`include` glob) and all imports in test
files that cross-reference helpers. Low urgency — tests run equally fast flat.

**D1–D4 — Dead exports in paths.ts / colors.ts:** `ASSET.favicon`, `ASSET.furnitureDefault`,
`AssetKey`, `shellBackground`, `gridBackground` are confirmed dead (no production import).
Safe to remove; deferred because the benefit is cosmetic and the risk is strictly mechanical.
